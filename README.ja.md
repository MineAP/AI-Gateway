# AI Gateway

> **今あるローカルAI環境を最大限に活かすための、シンプルで予測可能な OpenAI API 互換ゲートウェイ。**

AI Gateway は、最新のAIクライアントとローカルAI環境との相互運用性を向上させるための、軽量な OpenAI API 互換ゲートウェイです。

まずは Codex のような最新のAIクライアントと、LM Studio をはじめとするローカルLLM環境との互換性向上を目的としています。モデルごとの互換性の違いを吸収し、安定したインターフェースを提供することで、AIクライアントはバックエンドの違いを意識することなく利用できます。

AI Gateway 自身がAIとしてインフラを判断・制御することは目指していません。互換性変換やルーティングは明示的なルールに基づいて行い、ユーザーがAI環境の動作を予測し、制御できることを重視しています。

---

# Motivation

AI関連のソフトウェアは急速に進化しています。一方で、個人が所有するハードウェアは数年単位で更新されることが一般的です。

その結果、多くの開発者は次のような様々なリソースを組み合わせてAI環境を構築しています。

- デスクトップPCのGPU
- 常時稼働のホームサーバー
- ノートPC
- ローカルLLM
- MCPサーバー
- 検索サービス

しかし、それらを最新のAIクライアントから一貫した方法で利用するには、多くの場合モデル固有の調整や手作業による設定が必要になります。

AI Gateway は、こうした既存のローカルAI環境を OpenAI API 互換のインターフェースで統合し、最新のAIクライアントからより扱いやすい環境を提供することを目指しています。

より高性能なハードウェアを前提とするのではなく、**限られたローカルAIリソースを効率よく活用しながら、今ある資産を最大限に活かしてAIをより便利に利用できる環境を整えること**が、このプロジェクトの目的です。

# Design Principles

## Keep Models Simple

LLMが扱う Function Calling API はできるだけシンプルに保ちます。

バックエンドの実装が変わっても、プロンプトやツール定義を変更する必要がない設計を目指します。

---

## Backend Independence

Gateway API は、バックエンドの実装から独立したインターフェースを提供します。

例えば

```text
search_documentation()
```

というFunctionは、将来的に

- Docker MCP Toolkit
- ローカル検索
- RAG

など異なる実装へ置き換えられても、AIクライアント側の変更は不要です。

---

## Predictable by Design

AI Gateway は、自律的な判断よりも明示的な設定を重視します。

互換性変換、ルーティング、バックエンド連携は、AIによる判断ではなく、決定的なルールによって制御されます。

AI Gateway の役割は、AIの推論を置き換えることではなく、AIがローカルAI環境をより効率よく利用できるインフラを提供することです。

予測可能であることは使いやすさだけではありません。

**限られたローカルAIリソースを効率よく活用するための設計思想**でもあります。

---

## Small Start

AI Gateway は、小さく始めて段階的に成長することを前提に設計されています。

各フェーズでは、まず目の前の課題を最小限のアーキテクチャで解決し、その後必要に応じて機能を追加していきます。

Remote MCP、RAG、ワークフロー、自動ルーティングなどの機能も、十分な価値があると判断した段階で導入する方針です。

---

## Docker-first

各コンポーネントは独立したDockerコンテナとしてデプロイできることを基本方針とします。

これにより、デスクトップPC、ホームサーバー、将来の実行環境でも、一貫した構成で運用できることを目指します。

# High-Level Architecture

現在のAI Gatewayは、最新のAIクライアントとローカルAI環境を接続する軽量な互換レイヤーとして実装を進めています。

```text
                    AI Client
              (Codex, ChatGPT, etc.)
                        │
              OpenAI-compatible API
                        │
            +-------------------------+
            |       AI Gateway        |
            |-------------------------|
            | HTTP API Endpoint       |
            | Compatibility Pipeline  |
            | AI Provider Adapter     |
            +-------------------------+
                        │
                        ▼
                Local AI Environment
                  ├── AI Provider
                  │      ├── LM Studio
                  │      │      └── Local LLM
                  │      │
                  │      └── Future Providers
                  │
                  ├── MCP Server (将来)
                  ├── Search Service (将来)
                  └── RAG (将来)
```

初期実装では、AIクライアントとローカルLLMとの互換性向上に集中します。

その後も OpenAI API 互換インターフェースを維持しながら、必要に応じてバックエンドサービスを段階的に追加していきます。

詳細は **docs/architecture.md** を参照してください。

AI Gateway は、段階的な進化を前提として設計されています。

# なぜ決定的なルーティングを重視するのか

AIによる自動判断にもコストがあります。

適切でない判断が行われた場合、多くは「失敗 → リトライ」という形になり、そのたびに推論時間とGPUリソースを消費します。

クラウド環境では運用コストの増加に留まる場合でも、ローカルAI環境ではGPU時間・メモリ帯域・消費電力などの限られた計算資源を直接消費することになります。

そのためAI Gateway は、可能な限り明示的なルールに基づく決定的なルーティングを採用し、本当にAIの推論が必要な場面へ計算資源を集中できる設計を目指しています。

将来的にAIによる自動ルーティングが十分な価値を提供できると判断した場合には、その機能を段階的に導入する予定です。
